### Страница Library

[Pull Request: Library](https://github.com/NotDmitry/Duoscript-Tandem/pull/100)

Разработка всех компонентов происходила с использованием Material UI. Library представляет собой список всех доступных
для пользователя курсов и уроков и состоит из вложенных страниц. Основная страница _Library_ - это набор карточек с
курсами, содержащими прогресс пользователя по каждому курсу. Каждый курс ведёт на страницу _CourseLessons_ с
пронумерованным списком уроков данного курса и отметкой о выполнении урока. В свою очередь выбор урока перенаправляет
пользователя на страницу _Lesson_, которая представляет собой "движок" для запуска и отображения виджетов. Каждый урок
связан со своим виджетом.

### Backend-as-a-Service (BaaS)

[Pull Request: Data models](https://github.com/NotDmitry/Duoscript-Tandem/pull/69)  
[Pull Request: Preparations for migrating project to Firebase API](https://github.com/NotDmitry/Duoscript-Tandem/pull/86)  
[Pull Request: Firebase API](https://github.com/NotDmitry/Duoscript-Tandem/pull/94)

Для работы приложения с данными было принято решение использовать _Firebase_ как надёжный и быстрый способ организации
взаимодействия с сервером. Для маппинга данных между сервером и клиентом созданы модели типов данных, описывающих
сущности, хранимые в БД, а также их представления, нормализованные для извлечения необходимых данных и отображения их
на фронтенде. Были выделены следующие сущности:

- User - пользователь приложения;
- Course - курс по определённой теме;
- Lesson - урок, содержащий ссылку на виджет и используемый как часть курса;
- Widget - сложная структура, состоящая из общих данных об упражнении и объекта конфигурации, который имеет
  разное представление в зависимости от упражнения;
- Activity - информация о результатах пользователя во взаимодействии с упражнениями;
- CourseProgress - информация о прогрессе пользователя в рамках начатых им курсов.

Основной трудностью в разработке моделей выступил вскрывшийся в процессе факт того, что Firestore использует
нереляционную модель данных, основанную на документах и коллекциях, а это: проблемы с пагинацией, дублирование полей
для уменьшения количества запросов, невозможность объединять запросы к разным сущностям и др.

Для обеспечения параллельной разработки было принято решение тестировать работоспособность виджетов на моковых данных,
то есть предустановленных константных объектах. Впоследствии эта концепция была расширена до создания взаимозаменяемых
моковых и "живых" (_Firebase_) реализаций API-слоя, которые скрыты за единым интерфейсом, поэтому код-потребитель не
знает о деталях реализации, а переключение осуществляется настройкой переменных окружения.

Данный подход имеет следующие основания:

- снижение нагрузки на _Firebase_ во время разработки отдельной функциональности;
- возможность офлайн разработки в случае непредвиденной ситуации в сети или проблем с доступом к _Firebase_;
- дальнейшая разработка новой функциональности без опасения нарушения целостности существующих данных;
- параллельная разработка фронтенда и бэкенда.

В целом, знакомство с _Firebase_ как с инструментом быстрой настройки бэкенда оказалось полезным, но структура проекта,
API-слоя, моделей данных разрабатывалась с возможностью дальнейшего перехода на собственный полноценный бэкенд без
значительных изменений в потоке управления и данных.
