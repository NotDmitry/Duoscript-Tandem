<span style="display:inline-block;margin:0 0 8px 4px;font-size:18px;font-weight:700;color:steelblue">PROJECT DESCRIPTION</span><br>
<span style="font-size:16px;font-weight:700;padding:0;margin:0 0 0 14px">Введение</span>
<pre style="font-size:14px;font-weight:400;background:none;padding:0;margin:0 0 0 14px">
    Проект разрабатывался в рамках финального задания RS School TANDEM командой из
четырёх человек — Алексей, Дима, Света и Алёна — находящихся в разных часовых поясах: 
Будапешт, Минск и Калифорния. Для организации созвонов я составил таблицу с текущим 
временем каждой локации, что позволило быстро находить удобное общее время.
</pre>
<br>
<span style="font-size:16px;font-weight:700;padding:0;margin:0 0 0 14px">Стек и технологические решения</span>
<pre style="font-size:14px;font-weight:400;background:none;padding:0;margin:0 0 0 14px">
    В качестве основы выбрали React + Vite — на тот момент новый стек для большинства из нас. 
Backend реализован на Firebase: удобно, бесплатно и не требует инфраструктурных затрат.
Одним из ключевых решений стал переход на Material UI. Изначально несколько дней ушло 
на создание собственных атомарных компонентов и изучение подходов к построению UI-библиотеки.
После большого количества ресёрча и командного созвона, на котором я привёл аргументы, в том 
числе на основе доклада Cory House, команда приняла решение полностью переключиться на MUI.
Уже сделанное пришлось удалить, но в перспективе это существенно ускорило разработку.
</pre>
<br>
<span style="font-size:16px;font-weight:700;padding:0;margin:0 0 0 14px">Дизайн и Figma</span>
<pre style="font-size:14px;font-weight:400;background:none;padding:0;margin:0 0 0 14px">
    С самого начала я взял на себя дизайн проекта. Создал черновые макеты всех страниц: 
Landing, Dashboard, About, Library, Registration, Profile и 404. После перехода на 
MUI полностью перестроил все макеты в Figma с использованием MUI-библиотеки компонентов. 
Чтобы команда работала с компонентами единообразно, я записал и опубликовал туториал-видео.
</pre>
<br>
<span style="font-size:16px;font-weight:700;padding:0;margin:0 0 0 14px">Code review</span>
<pre style="font-size:14px;font-weight:400;background:none;padding:0;margin:0 0 0 14px">
    На протяжении всего проекта активно участвовал в ревью кода: предложил упростить сложный
Regex коллеги с негативного паттерна до одного позитивного условия, отметил нерекомендуемые
практики TypeScript в другом PR, поднял вопрос о непоследовательном расположении типов и 
интерфейсов по файлам.
</pre>

<span style="font-size:16px;font-weight:700;padding:0;margin:0 0 0 14px">Landing Page</span>
<pre style="font-size:14px;font-weight:400;background:none;padding:0;margin:0 0 0 14px">
    Страница реализована строго по макету Figma и почти не содержит логики. 
В её состав вошли: приветственная строка, которая динамически меняется в зависимости
от того, кто открывает страницу — гость или авторизованный пользователь;
описание приложения, взятое из README.md репозитория проекта; два изображения-превью, 
демонстрирующих возможности приложения; и CTA-кнопка, ведущая на страницу 
регистрации — она отображается только для гостей. Страница получилась лаконичной и 
не перегруженной.
</pre>
<br>
<span style="font-size:16px;font-weight:700;padding:0;margin:0 0 0 14px">About Page</span>
<pre style="font-size:14px;font-weight:400;background:none;padding:0;margin:0 0 0 14px">
    Страница по своей природе несложная — она практически не содержит логики и состоит из UI.
Главной задачей при её создании было написать текст описания приложения: достаточно ёмкий,
чтобы не перегрузить пользователя, и достаточно понятный, чтобы с первого взгляда было ясно,
для чего служит приложение. Подход оказался простым: я проговаривал описание вслух, как сам 
его понимаю, записывал в блокнот, причёсывал текст и переводил на английский с помощью Google.
</pre>
<br>
<span style="font-size:16px;font-weight:700;padding:0;margin:0 0 0 14px">Error Page</span>
<pre style="font-size:14px;font-weight:400;background:none;padding:0;margin:0 0 0 14px">
    Для страницы 404 хотелось чего-то простого, но запоминающегося — получилась страница с 
покачивающимся Sponge Bob. Анимацию через SCSS или inline-стили делать не хотелось. 
Решением стал keyframes из @emotion/react — анимация осталась прямо рядом с компонентом 
и не потребовала отдельного файла.
</pre>
<br>
<span style="font-size:16px;font-weight:700;padding:0;margin:0 0 0 14px">Widget Quiz</span>
<pre style="font-size:14px;font-weight:400;background:none;padding:0;margin:0 0 0 14px">
    Виджет Quiz написан с нуля. Начал с реализации только под CSS-задачи, но быстро увидел
возможность сделать виджет универсальным: через `type` prop один и тот же компонент стал
работать для HTML, CSS, JavaScript, TypeScript и GitHub-тематик — по 45–47 вопросов в 
каждой группе (для peer review в последствии сократил до пяти). В процессе освоил
useState, useEffect и useRef. 
Главной технической сложностью была работа с асинхронными данными внутри кастомного хука: 
решение — объявить async-функцию внутри `useEffect` и передавать результат через useState.
Для этого виджета написал юнит-тесты на Vitest — впервые в жизни.
</pre>
<br>
<span style="font-size:16px;font-weight:700;padding:0;margin:0 0 0 14px">Widget Bug Hunter</span>
<pre style="font-size:14px;font-weight:400;background:none;padding:0;margin:0 0 0 14px">
    Bug Hunter оказался самым сложным с точки зрения архитектуры данных. Каждое задание 
содержит код, в котором от одного до четырёх мест нужно исправить через выпадающие списки.
Главная проблема — как точно указать, в каком месте строки кода должен быть вставлен dropdown.
Решение получилось прямолинейным, но работающим: я ввёл маркер `XDX` — литеральный
placeholder прямо в строке кода. При рендеринге строка разбивается методом .split('XDX'), 
и между фрагментами вставляются списки с вариантами ответов. Виджет охватывает JavaScript, 
HTML, CSS, TypeScript и GitHub.
</pre>
